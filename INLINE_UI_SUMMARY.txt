2. HOW IS IT STRUCTURED (HTML/DOM)?
   - Dynamic HTML container created by JavaScript
   - Structure:
     <div id="bythehour-inline-control">
       <span>Posted within</span>
       <input type="number" id="bth-hours" min="1" value="12">
       <span>hours</span>
       <button id="bth-save">Apply</button>
       <span id="bth-status"></span>
     </div>
   - Uses inline-flex layout to align components horizontally
   - Responsive design with compact mode for narrow screens
   - Positioned next to the location filter button in the results header

3. HOW DOES USER ADJUST HOURS THRESHOLD?
   - Two methods:
     a) Click the "Apply" button (click event listener)
     b) Press Enter key while focused on input (keydown event listener)
   - Both methods trigger handleSaveFromUi() function
   - Input validation ensures only positive integers are accepted
   - On success: button shows "Saved" (green) for 800ms, then reverts
   - On failure: error message displayed in red

4. WHAT EVENT LISTENERS/HANDLERS ARE ATTACHED?
   - Input element: keydown listener (Enter key)
   - Save button: click listener
   - Global: chrome.storage.onChanged listener (syncs changes from other tabs/popup)
   - Global: MutationObserver on document.body (re-runs filter when page changes)
   - All listeners prevent duplicate attachment via dataset.bound check

5. HOW DOES INLINE UI INTERACT WITH chrome.storage?
   - Uses chrome.storage.sync (cross-device, cross-browser sync)
   - Load: loadMaxHours() reads stored value, falls back to 12 hours
   - Save: saveMaxHours(value) writes new threshold
   - Listen: chrome.storage.onChanged fires when ANY source changes the value
   - This enables automatic sync between popup and inline UI
   - Both have error handling and fallback values

   - Keydown listener on input field (Enter only) also calls handleSaveFromUi()

4. User Interaction Handler (Lines 905-938):
   - handleSaveFromUi() coordinates entire save flow
   - Validates input via parseHoursInput()
   - Disables button during save
   - Calls saveMaxHours(parsed) to persist to storage

5. Storage Integration (Lines 301-341):
   - loadMaxHours(): Promises-based wrapper around chrome.storage.sync.get
   - saveMaxHours(): Promises-based wrapper around chrome.storage.sync.set
   - Both have extension context checks and error handling
   - Fall back to DEFAULT_MAX_HOURS (12) on any error

6. Storage Listener (Lines 1017-1034):
   - chrome.storage.onChanged fires when value changes from ANY source
   - Updates input field value
   - Syncs status message
   - Immediately re-runs filter
   - This creates real-time sync between popup and inline UI

7. UI Mount Target Detection (Lines 575-589):
   - Tries to find location button in results header
   - Returns mount target with strategy and mode
   - Allows flexible positioning as LinkedIn layout changes
   - Returns object with { mode, node, containerNode, strategy }

8. DOM Attachment (Lines 591-626):
   - attachInlineControl() handles three attachment modes
   - "after": Inserted after location button (sibling insertion)
   - "append": Appended as child of container
   - "prepend": Inserted as first child
   - Checks position to avoid re-inserting if already correct

9. Retry Logic (Lines 702-724):
   - scheduleUiRetryLoop() keeps trying to mount UI
   - Retries every 1.2 seconds up to 120 times (≈2.4 minutes)
   - Stops if UI becomes visible or max attempts reached
   - Syncs value from storage on first successful mount

10. Input Validation (Lines 896-903):
    - parseHoursInput() ensures valid positive integer
    - Returns null for invalid input
    - Floors decimal values
    - Returns null for values < 1

STORAGE FLOW:

Save Flow:
  User input → handleSaveFromUi() → parseHoursInput() → saveMaxHours()
  → chrome.storage.sync.set() → Storage updated

Notification Flow:
  Storage updated → chrome.storage.onChanged (all tabs)
  → Update UI value + status + runFilter()

Load Flow:
  syncUiFromStorage() → loadMaxHours() → chrome.storage.sync.get()
  → Parse and validate → Update UI input field
